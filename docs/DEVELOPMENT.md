# Development Guide

## Requirements

- Python 3.11+
- Node.js 24+
- npm 11+
- PowerShell

## Setup

From the repository root:

```powershell
python -m venv .venv
.\.venv\Scripts\python -m pip install --upgrade pip
.\.venv\Scripts\python -m pip install -e .\backend[dev]
cd frontend
npm install
npx playwright install chromium
cd ..
```

## Run The Dev Stack

```powershell
.\scripts\dev.ps1
```

The script starts:

- FastAPI on the first free backend port starting at `8000`.
- Vite on `0.0.0.0:5173` for local-network browser access.
- World watching enabled.
- Uvicorn reload enabled for backend code changes.
- A generated unlock code if `VIRTUALSCREEN_ACCESS_TOKEN` is not already set.

The script prints:

- Local DM URL: `http://localhost:5173`
- LAN DM URL: `http://<dm-pc-lan-ip>:5173`
- Unlock code
- PID file path

Stop the stack:

```powershell
.\scripts\stop-dev.ps1
```

Check expected dev and e2e ports:

```powershell
.\scripts\check-dev.ps1
```

## Verification

Run the fast V1 verification gate:

```powershell
.\scripts\test.ps1
```

It runs, in order:

1. E2E port cleanup and port report.
2. Release hygiene checks for sample-world/runtime artifacts.
3. Backend pytest.
4. Backend Ruff checks.
5. Frontend Vitest.
6. Frontend production build.
7. Playwright smoke e2e tests.
8. E2E port cleanup and a stage summary.

Run the exhaustive browser suite when preparing a release candidate:

```powershell
.\scripts\test.ps1 -E2E full
```

Skip browser tests only while narrowing a non-browser failure:

```powershell
.\scripts\test.ps1 -E2E none
```

Focused commands:

```powershell
.\.venv\Scripts\python -m pytest backend
.\.venv\Scripts\python -m ruff check backend
cd frontend
npm run test
npm run build
npm run test:e2e:smoke
npm run test:e2e
```

If Playwright reports that `5174` or `8010` is already in use, run:

```powershell
.\scripts\check-dev.ps1
.\scripts\stop-dev.ps1
.\scripts\check-dev.ps1
```

If the port is still owned by another process, stop that process before retrying.

Run the hygiene gate directly when preparing a release diff:

```powershell
.\scripts\release-hygiene.ps1
```

The sample world should contain tiny owned fixtures only. Runtime files under
`sample-world/.virtualscreen` are release blockers except for intentional JSON card
templates in `sample-world/.virtualscreen/card-templates`.

## Useful Environment Variables

- `VIRTUALSCREEN_WORLD_ROOT`: initial active world folder.
- `VIRTUALSCREEN_WORLDS_ROOT`: managed world library folder.
- `VIRTUALSCREEN_ACCESS_TOKEN`: LAN unlock code; generated by `dev.ps1` when empty.
- `VIRTUALSCREEN_WATCH_WORLD`: enable live world watching.
- `VIRTUALSCREEN_API_TARGET`: frontend dev proxy backend target.

## Manual Smoke Targets

- DM workspace: `http://localhost:5173`
- Public player screen: `http://localhost:5173/screen`
- Backend health: `http://localhost:<backend-port>/api/health`
- Prep Check: workspace controls > `Prep Check` > `Run Check`

When auth is enabled, the DM workspace and DM APIs require unlock. The player screen is
public but can only read currently displayed content and embedded media dependencies.

Prep Check is explicit and on-demand. It rebuilds the current index for the audit and
reports unresolved links, missing embeds, and literal saved `.dms` command paths; it does
not watch continuously or scan raw HTML media.

## DMS Scripts

Normal DM automation uses trusted `.dms` files stored anywhere in the active world. Open a
script tab, edit and save it, then run it from the editor toolbar, Scripts tool, or a Run
script fast slot. Temporary Markdown/CSV output tabs can be saved back to the world with
Save As.

Legacy deprecated `/api/scenarios` endpoints remain available for compatibility tests and
old integrations, but new frontend work should target `.dms` scripts.

## Coding Notes

- Keep all world access behind safe path helpers.
- Do not store user-authored source content only in SQLite.
- Keep `.virtualscreen` internals hidden from normal world browsing.
- Add focused tests before changing path safety, indexing, editing, display, auth, or
  DMS script execution behavior.
